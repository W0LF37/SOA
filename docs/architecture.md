# System Architecture вҖ” AI Project Manager

## Overview

AI Project Manager is a research-oriented planning system that converts project requirements into executable software project artifacts. It accepts either narrative project briefs or structured requirement templates, parses them into normalized requirement items, generates a task plan, estimates effort, validates quality, constructs dependency relationships, analyzes delivery risk, and exports machine-readable planning outputs.

The system is designed around local-first AI and reproducible evaluation. Ollama/Qwen provides private LLM-guided planning when available, while deterministic rule-based validators preserve reliability for tests and offline execution. A ChromaDB knowledge base supplies retrieval-augmented planning examples and estimation patterns, and graph analytics convert a flat task list into a dependency-aware delivery plan suitable for sprint planning and risk review.

The intended architecture is hybrid rather than fully model-trained. The LLM acts as a Planning Reasoner over normalized requirements and retrieved KB context. Rule-based logic remains as the safety layer that validates JSON shape, source references, FR/NFR classification, complexity bounds, and dependency integrity before any plan is accepted.

## LLM And KB Traceability

Each pipeline run records the Planning Reasoner outcome in `plan_summary.json`: whether the LLM was attempted, whether its plan was accepted, the fallback reason when rules were used, the KB document count, and the retrieved KB context that was injected into the LLM prompt. This makes the AI role auditable instead of hidden behind a generic "generated by AI" label.

## Knowledge Base Contents

The ChromaDB collection contains curated records in four main groups: historical estimation records, COCOMO calibration points, task-level estimation patterns, and planning examples. Planning examples guide the LLM on task decomposition, FR/NFR classification, and complexity levels. Estimation patterns calibrate the EffortEstimator through RAG without requiring a custom-trained regression model.

## Component Decisions

| Component | Technology | Why Chosen | Alternative Considered |
|---|---|---|---|
| Parser | Custom Python | Domain flexibility | spaCy NLP (too heavy) |
| LLM | Ollama/Qwen | Local Planning Reasoner with rule-validated output | OpenAI API (paid, online) |
| Vector DB | ChromaDB | Embedded RAG store for planning and estimation context | Pinecone (cloud only) |
| Graph | NetworkX | Python-native DAG | Neo4j (separate server) |
| UI | Streamlit | Rapid prototyping | React (too complex for research) |

## Evaluation Methodology

The evaluation suite measures each sample against expected task counts, classification behavior, coverage, complexity distribution, and graph validity. The primary metrics are normalized to the range 0.0 to 1.0.

1. Requirement coverage:
   `coverage = covered_requirements / total_requirements`

   A requirement is considered covered when at least one generated task preserves its core intent. Higher coverage indicates fewer missed requirements.

2. Classification score:
   `actual_fr_ratio = fr_count / task_count`

   `classification = max(0.0, 1.0 - abs(actual_fr_ratio - expected_fr_ratio) / 0.3)`

   This measures how close the generated FR/NFR distribution is to the expected distribution for a sample. The denominator `0.3` provides tolerance around the expected ratio.

3. Complexity calibration:
   `complexity = min(1.0, unique_complexity_values / 3.0)`

   This penalizes plans where all tasks have the same complexity. A well-calibrated plan should contain a small but meaningful range of complexity levels.

4. Dependency coherence:
   `dependency = max(0.0, 1.0 - validation_warning_count * 0.1)`

   DependencyGraph validation checks for structural problems such as cycles or suspicious graph patterns. Each warning reduces the score by 0.1.

5. Overall system score:
   `overall = mean(coverage, classification, complexity, dependency)`

   This combines semantic completeness, requirement classification quality, estimation calibration, and graph quality into one research summary metric.
